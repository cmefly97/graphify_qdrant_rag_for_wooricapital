왜 만들었나

  • 심사·운영 기준이 3종 출처(FAQ·규정·월별 운영기준)에 흩어져 있고 수치가 매달 변동
  • 상담사가 정확한 금리·등급·한도를 빠르게 찾기 어려움
  • 일반 LLM/RAG는 정밀 수치에서 환각 — 금융 상담엔 치명적("21.0%"가 "20.0%"만 돼도 사고)

해결 원칙 — 환각 없는 상담

① 수치 환각 금지

금리·등급·금액은 구조화 테이블(SQLite) 조회 결과로만 생성. LLM 자유생성 금지.

② 출처 의무

모든 답변에 근거(문서·조항·유효일자)를 첨부. 근거 없는 주장 금지.

③ 모르면 모른다

근거 없으면 "규정에 명시되어 있지 않습니다". 추측 금지.

설계 결정 (ADR-001 / 006)

왜 Neo4j 대신 Qdrant + graphify인가

Neo4j (풀 그래프 DB)

  • 수백만 노드·복잡한 다중홉 Cypher에 강점
  • 별도 DB 서버 운영(메모리·백업·인프라 승인)
  • 단일 공유 그래프DB → 재인덱싱·동시쓰기 시 공유DB 충돌(락)
  • 현 데이터(57노드)엔 과잉 — 속도 이점 0

선택 — Qdrant + NetworkX + graphify

  • graphify가 만든 graph.json을 메모리 로드 → 즉시 활용
  • 수치=SQLite · 의미=Qdrant · 관계=NetworkX 분업
  • 운영 단순·경량, 사내 인프라 부담 최소
  • 확장 트리거: 노드>2만 또는 멀티홉 지연 시 Neo4j/AGE 재검토

  • nodes: label · file_type(document/concept) · source_file · community
  • links: relation(references / semantically_similar_to) · confidence(EXTRACTED/INFERRED)
  • hyperedges · 커뮤니티 9 · God node · betweenness

핵심 ② 수치 환각 방지

구조화 테이블(SQLite) = 정답의 단일 출처

  • 운영기준의 금리·조건을 텍스트가 아닌 구조화 데이터로 추출(병합 셀 처리)
  • LLM이 숫자를 만들지 않고 행에서 직접 조회 → 환각 원천 차단
  • source 전 문서를 훑어 누락·오류를 반영(NICE 등급표 실제값 교체 포함)

왜 빠른가 = 왜 정확한가

수치형은 테이블 직조회라 생성 0ms·환각 0. 벡터·그래프(④⑤)는 답이 아니라 근거 표시용, 지연 대부분은 임베딩 호출.

조건 대조 후 가능/불가 판정

Q. 나이스 955점, KCB 800점인데 프로모션 사용 가능해?

A. 아니요, 불가능합니다

프로모션 G/L 7.6%는 금리등급 1등급 & NICE 933↑ & KCB 976↑ 세 조건을 모두 충족해야 적용됩니다(법인 제외).

동작

  • 질의에서 NICE·KCB 점수 파싱
  • 기준값(933·976)은 tables.db에서 자동 추출(하드코딩 X)
  • 조건별 충족/미충족 판정 + 근거 제시

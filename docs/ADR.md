# ADR — 왜 이렇게 만드는가 (Architecture Decision Records)

> 형식: 결정 / 배경 / 대안 / 근거 / 결과

## ADR-001 저장소: 경량 Qdrant + NetworkX
- **결정**: 그래프는 NetworkX(graph.json 메모리 로드), 벡터는 Qdrant.
- **배경**: 현재 코퍼스 ~4,872단어 / 57노드. graphify 리포트도 "그래프 불필요할 수 있음" 경고.
- **대안**: Neo4j+벡터인덱스 / Postgres(pgvector+AGE).
- **근거**: 이 규모에서 속도 차이 무의미, Neo4j는 과잉 운영부담. graphify 산출물이 이미 graph.json이라 즉시 활용 가능.
- **결과**: 구축 단순·운영 경량. 데이터가 수만 노드로 커지면 Neo4j/AGE 재검토(트리거: 노드>20k 또는 멀티홉 지연 체감).

## ADR-002 수치·규칙: 구조화 테이블 + 하이브리드 라우팅
- **결정**: 정밀 수치는 별도 구조화 테이블 조회, 서술형은 그래프RAG, 라우터가 분기.
- **배경**: 질의 다수가 금리·등급·금액 등 정밀값. 순수 RAG는 수치 환각 위험.
- **대안**: 순수 그래프+벡터 RAG(텍스트 청크만).
- **근거**: 상담 도구는 숫자 1개 오류도 사고. 테이블 조회는 환각 원천 차단. 라우팅으로 서술형 유연성도 확보.
- **결과**: 정확도↑, 대신 표 추출·스키마·라우팅 로직 비용 추가. (수치형 답변의 정본은 tables.db)

## ADR-003 배포: POC + 운영확장 구조
- **결정**: 단일 서버 POC로 시작하되 인증·로깅·재인덱싱을 빈 인터페이스로 분리.
- **배경**: 단순 데모가 아니라 다양한 질문에 답하는 MVP 목표.
- **대안**: 순수 데모 / 처음부터 풀 운영.
- **근거**: 빠른 가치검증 + 재작성 없는 운영 전환의 균형.
- **결과**: MVP 빠르게, 후속에 모듈 끼움.

## ADR-004 LLM/임베딩: 사내 게이트웨이 전용
- **결정**: HCX-30B-Text(`hcx-agent-05`, 223.130.140.68:11000) / Qwen3.6-35B-A3B 폴백(223.130.140.218:8000) / Octen-Embedding-8B(223.130.140.218:8002)만 사용.
- **근거**: 사내 보안·규정 준수, 모델 관리 시스템 일원화.
- **결과**: 외부 모델 금지. 키는 .env. HCX 장애 시 Qwen 폴백.

## ADR-005 UI: 채팅 + 근거 그래프 시각화
- **결정**: 답변 옆에 근거 노드/출처를 표시.
- **근거**: 상담사 신뢰 확보 + 개발 디버깅. 환각·오답을 사람이 즉시 검증.
- **결과**: Cytoscape.js 패널 추가 구현.

## ADR-007 운영 전환: 2·3·5 먼저, 4·1·6 보류
- **결정**: 로깅·대시보드(2), 자동 재인덱싱(3), CI 게이트·A/B(5)를 먼저 구현. 인프라(4)·SSO(1)·보안(6)은 문서화 후 보류.
- **근거**: 4·1·6은 사내 인프라/SSO/보안 정책 사양에 강하게 의존 → 사양 확정 전 구현은 재작업 위험. 2·3·5는 코드 내에서 자립적으로 구현·검증 가능하고 품질 운영의 토대.
- **결과**: user_id 는 SSO 전까지 'anonymous'. 스케줄러 실행은 사내망. 상세 docs/PRODUCTION_TRANSITION.md.

## ADR-008 업그레이드 회귀는 A/B(베이스라인 비교)로 관리
- **결정**: 골든셋 평가 결과를 baseline.json 으로 스냅샷하고, 변경 시 현재 vs 베이스라인을 비교(ab_test)해 항목별 개선/회귀를 본다. ci_gate 가 임계 미달을 차단.
- **근거**: 단일 정확도 수치만으론 "무엇이 좋아지고 무엇이 깨졌는지" 안 보임. 항목 단위 A/B 가 회귀를 조기 포착.
- **결과**: 모든 업그레이드 후 `ab_test` 권장. 테스트 코드는 파일로 보존.

## ADR-006 graphify는 오프라인 빌드에만 사용
- **결정**: graphify(safishamsi/graphify)는 인덱싱 단계의 그래프 추출에만 쓰고, 서빙은 자체 하이브리드 엔진.
- **근거**: graphify는 "폴더→지식그래프" 추출 도구이지 실시간 서빙엔진이 아님.
- **결과**: graph.json을 산출물로 소비, 노드↔청크 매핑으로 연결.

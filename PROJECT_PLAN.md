# 우리캐피탈 오토운영팀 상담챗봇 — 프로젝트 계획서

> 그래프RAG + 벡터 임베딩 하이브리드 검색 기반 사내 상담 지원 시스템
> 작성일: 2026-06-28 · 수준: MVP · 단계: 설계 확정

---

## 1. 목표와 범위

### 1.1 목적
우리캐피탈 오토운영팀 상담사가 **중고오토리스·중고승용·재고금융·중형트럭 등 심사/운영 기준**에 대해 질문하면, 사내 규정·FAQ·월별 운영기준을 근거로 **정확하고 출처가 명확한 답변**을 제공하는 챗봇.

핵심 성공 기준은 두 가지다.
- 예시 질문은 물론, **예시에 없는 다양한 변형 질문**에도 의미 기반으로 답한다.
- 금리·등급·금액 같은 **정밀 수치는 절대 지어내지 않는다.** 근거가 없으면 "규정에 명시되어 있지 않습니다"라고 답한다.

### 1.2 확정된 의사결정 (MVP)
| 항목 | 선택 | 의미 |
|---|---|---|
| 저장소 스택 | **경량: Qdrant(벡터) + NetworkX(그래프)** | graphify의 `graph.json`을 메모리에 로드, 운영 부담 최소 |
| 배포 범위 | **POC + 운영확장 구조 (옵션 2)** | 지금은 단일 서버, 인증·로깅·재인덱싱은 인터페이스만 비워두고 후속 연결 |
| 수치·규칙 처리 | **하이브리드 자동판단 (옵션 2)** | 질의를 분류해 수치형→테이블 조회, 설명형→그래프RAG 라우팅 |
| UI | **채팅 + 근거 그래프/출처 시각화 (옵션 2)** | 답변 신뢰성·디버깅을 위해 근거 노드/출처 표시 |

### 1.3 데이터 현황 (분석 결과)
- 출처 3종: **FAQ**(지점 일반문의, xlsx 52행) / **규정**(핵심 내부규정, docx·md) / **운영기준**(월별 금리·운영기준, pdf·docx)
- 파싱 대상 형식: doc/docx, pdf, md, xlsx, **이미지**(FAQ 답변 일부가 "이미지 참조")
- graphify 1차 산출물: **57노드 · 66엣지 · 9커뮤니티** (`graphify-out/graph.json`, NetworkX node-link 포맷)
- ⚠️ **중요 트레이드오프**: graphify 리포트가 직접 지적 — *"코퍼스가 ~4,872단어로 단일 컨텍스트에 들어감 → 그래프가 꼭 필요하지 않을 수 있음"*. 현재 데이터 규모에선 그래프RAG의 멀티홉 이점보다 **정밀 수치·규칙 조회 정확도**가 더 중요하다. 따라서 본 설계는 그래프RAG를 "관계 탐색·근거 보강"에, 구조화 테이블을 "정밀 수치 조회"에 쓰는 하이브리드로 간다. 데이터가 늘어나면 그래프 비중이 자연스럽게 커진다.

## 2. 시스템 아키텍처

### 2.1 전체 구성
```
┌─────────────────────────────────────────────────────────────┐
│  Frontend (React/Next.js)                                    │
│   · 예시 질문 칩  · 채팅창  · 답변+근거출처  · 근거 그래프 시각화  │
└───────────────┬─────────────────────────────────────────────┘
                │ REST / SSE(스트리밍)
┌───────────────▼─────────────────────────────────────────────┐
│  Backend API (FastAPI, Python)                               │
│                                                              │
│  [1] 질의 라우터(Query Router)                                 │
│        └ LLM 분류: 수치형 / 설명형 / 혼합형                      │

│  [5] 답변 생성(LLM) — 근거 인용 + "모르면 모른다"                 │
└───────────────┬──────────────────────────┬──────────────────┘
                │                          │
     ┌──────────▼─────────┐     ┌──────────▼──────────┐

     │ (namc-aigw.io.naver) │     │  · Qdrant(벡터)      │
     │ · HCX-30B(hcx-agent-05)│   │  · graph.json(NX)    │
     │ · Qwen3.6            │     │  · tables.db(수치)   │
     │ · Octen-Embedding-8B │     │  · docstore(원문청크) │
     └──────────────────────┘     └─────────────────────┘
                ▲
     ┌──────────┴──────────────────────────────────────┐
     │  오프라인 인덱싱 파이프라인 (배치)                    │
     │  파싱 → 청킹 → 엔티티/관계 추출(graphify) →           │
     │  임베딩 → Qdrant 적재 + 그래프 빌드 + 테이블 추출      │
```

### 2.2 두 갈래 데이터 경로
1. **오프라인(인덱싱)**: 문서를 받아 검색 가능한 인덱스로 변환. 월 1회 또는 운영기준 갱신 시 재실행.
2. **온라인(질의)**: 상담사 질문 → 라우팅 → 검색 → 답변. 실시간.

## 3. 데이터 인덱싱 파이프라인 (오프라인)

### 3.1 문서 파싱 (형식별 전략)
| 형식 | 도구 | 비고 |
|---|---|---|
| docx | `python-docx` / `mammoth` | 표·제목 구조 보존 |
| pdf | `pymupdf(fitz)` + 표는 `pdfplumber` | 운영기준 금리표 정확 추출이 관건 |
| md | 직접 파싱 | 5_재고금융은 이미 md |
| xlsx | `openpyxl` | FAQ는 구분/질문/답변 구조 그대로 Q-A 쌍으로 |
| 이미지 | **VLM(VLM 비전) OCR/캡셔닝** | "이미지 참조" 답변·표 이미지 텍스트화 |

> FAQ xlsx의 답변에 포함된 `<br>`, "(이미지 참조)" 등은 전처리 단계에서 정규화. 이미지 참조 항목은 원본 이미지를 VLM으로 텍스트화해 답변에 합친다.

### 3.2 청킹 전략
- 규정/운영기준: **조항·표 단위 의미 청킹**(고정 토큰 분할 지양). 표는 행 단위로 쪼개되 헤더를 각 청크에 반복 부착해 맥락 유지.
- FAQ: **질문-답변 1쌍 = 1청크** (가장 검색 친화적).
- 각 청크에 메타데이터 필수 부착: `source_file`, `doc_type(FAQ/규정/운영기준)`, `상품(중고리스/중고승용/재고금융/중형트럭…)`, `유효일자(예: '26.03)`, `조항/행 위치`.

### 3.3 그래프 빌드 (graphify 활용)
- 이미 생성된 `graphify-out/graph.json`을 그대로 1차 그래프로 사용.
- 노드: `{label, file_type(document/concept), source_file, community, id}`
- 엣지: `{relation(references/semantically_similar_to…), confidence, confidence_score, source, target}`
- 각 그래프 노드를 해당 원문 청크 및 임베딩과 **id로 연결**(노드↔청크 매핑 테이블). 그래야 그래프 탐색 결과를 실제 근거 텍스트로 환원 가능.
- ⚠️ graphify 리포트가 짚은 **고립 노드 18개·약연결 노드 20개**는 인덱싱 후 점검 대상. 누락된 관계(예: `채권서류 징구기준` ↔ 관련 상품)를 수동/LLM으로 보강하면 탐색 품질이 오른다.

### 3.4 임베딩
- 모델: **Octen-Embedding-8B** (사내 게이트웨이 `/v1/embeddings`)
- 청크 텍스트 + 핵심 메타(상품·유효일자)를 함께 임베딩해 의미+속성 동시 매칭.
- Qdrant 컬렉션에 `vector + payload(메타데이터)` 저장. payload 필터(상품·유효일자)로 사전 필터링 후 유사도 검색 → 정확도↑.

### 3.5 구조화 테이블 추출 (수치형 답변의 핵심)
운영기준의 **금리표·등급표·조건표**를 텍스트가 아닌 구조화 데이터로 별도 추출:
- 예: `금리표(상품=론/할부, 금리등급=2, GL금리=21.0%, nego규칙={거점장:11.0, 증빙:15.0, HJ:18.0}, NICE네고=1%, 유효일자='26.03)`
- 예: `Dual등급(상품=Dual_C, 내국인=7, 외국인=7)`
- 저장: SQLite 또는 parquet (`tables.db`). 질의 시 LLM이 자연어→구조화 파라미터로 변환해 정확 조회.
- 이 경로가 "금리등급 2등급 → 21.0%" 류 답변의 **환각을 원천 차단**한다.

## 4. 하이브리드 질의 처리 흐름 (온라인)

요청하신 흐름을 그대로 반영하고 라우팅을 추가했다.

```
사용자 질의: "로그인 튕김 현상" / "금리등급 2등급 최저금리?"
        │
[0] 질의 라우터 (LLM 분류)
        ├─ 수치형 ──────────────┐
        ├─ 설명형 ──────┐       │
        └─ 혼합형 ──────┤       │
                        ▼       ▼
[1] 쿼리 확장(LLM)    [3-A] 구조화 조회
   "인증 실패",         · 자연어→파라미터
   "세션 만료",         · tables.db 정확 조회
   "Auth fail" …       · 결과: 21.0% + nego규칙
        │
[2] 벡터 유사도 검색(Qdrant)
   키워드 달라도 '의미' 유사한
   [Session Timeout] 노드 발견 (시작점)
        │
[3-B] 그래프 탐색(NetworkX)
   시작 노드 → 인접 [OAuth2], [401], 같은 커뮤니티 노드 추적
   (1~2홉 확장 + 커뮤니티 컨텍스트)
        │
[4] 컨텍스트 조립 + 재랭킹
   벡터점수 + 그래프거리 + 메타필터(유효일자 최신)로 정렬
   각 근거에 출처(문서·조항·일자) 부착
        │
[5] 답변 생성(LLM, HCX-30B / hcx-agent-05)
   · 근거 기반 답변 + 출처 인용
   · 근거 없으면 "규정에 명시되어 있지 않습니다"
   · 수치는 [3-A] 결과를 우선 신뢰
        │
        ▼
   답변 + 근거출처 + 근거 그래프(UI 시각화)
```

### 4.1 라우팅 규칙(초안)
- **수치형**: 금리/등급/개월수/금액/한도/판정값 등 정확한 값 요구 → [3-A] 우선, [3-B]로 설명 보강.
- **설명형**: "가능해?", "어떻게 처리해?", 조건/절차 질문 → [3-B] 그래프RAG.
- **혼합형**: 둘 다 실행 후 통합 (예: "엔카 슬라이딩 가능해?" → 가능여부(규칙)+조건설명).

### 4.2 안전장치
- 답변 신뢰도 임계값 미만이면 "확실치 않음" 표기 + 관련 규정 위치 안내.
- 모든 답변에 **출처 메타(문서명·조항·유효일자)** 의무 표기 → 상담사가 원문 검증 가능.
- 유효일자가 다른 운영기준이 충돌하면 **최신본 우선** + 과거본 존재 알림.

## 5. 기술 스택

| 영역 | 선택 | 비고 |
|---|---|---|
| Backend | **Python 3.11 + FastAPI** | graphify·파싱 생태계가 Python |
| 벡터DB | **Qdrant** | 도커 단일 컨테이너, payload 필터 강력 |
| 그래프 | **NetworkX** (graph.json 로드) | 데이터 규모상 메모리로 충분 |
| 구조화 | **SQLite / parquet** | 금리·등급 테이블 |
| 문서스토어 | 파일/SQLite | 원문 청크 보관 |
| LLM(답변) | **HCX-30B-Text (`hcx-agent-05`)** | `HCX30_MODEL` |
| LLM(대안) | **Qwen3.6-35B** | 게이트웨이 재사용, 폴백/비교용 |
| 임베딩 | **Octen-Embedding-8B** | `/v1/embeddings` |
| 이미지 | **VLM 비전** | OCR/캡셔닝 |
| Frontend | **Next.js + React** | 채팅 UI |
| 그래프 시각화 | **Cytoscape.js** 또는 react-force-graph | 근거 노드 표시 |
| 설정 | **.env** (제공된 변수) | 키·엔드포인트·모델명 분리 |
| 배포 | **Docker Compose** | API + Qdrant, 운영전환 용이 |

## 6. UI 설계 (옵션 2: 채팅 + 근거 시각화)

- **예시 질문 칩**: 상단에 4~6개 노출 (제공된 예시: 취급 개월수, 금리 조회, Dual 등급, 엔카 슬라이딩, R판정 등).
- **채팅 영역**: 질문 입력 → 스트리밍 답변(SSE).
- **답변 카드**: 본문 + 하단 "근거" 섹션(문서명·조항·유효일자, 클릭 시 원문 청크 펼침).
- **근거 그래프 패널**(접이식): 이번 답변이 거친 노드/관계를 작은 그래프로 시각화 → 신뢰·디버깅.
- **수치 답변 강조**: 표 형태로 금리/등급/nego를 구조적으로 표시.

## 7. 단계별 로드맵

### Phase 0 — 셋업 (0.5주)
- 레포 구조, Docker Compose(API+Qdrant), .env 로딩, 게이트웨이(LLM·임베딩) 연결 스모크 테스트.

### Phase 1 — 인덱싱 파이프라인 (1.5주)
- 형식별 파서(docx/pdf/md/xlsx/이미지) → 청킹 → 메타 부착.
- Octen 임베딩 → Qdrant 적재. graph.json 로드·노드↔청크 매핑.
- 운영기준 금리·등급 테이블 구조화 추출 → tables.db.
- 산출 점검: 고립/약연결 노드 보강.

### Phase 2 — 질의 엔진 (2주)
- 질의 라우터(분류) + 쿼리 확장.
- 하이브리드 검색(벡터→그래프 탐색) + 재랭킹 + 출처 부착.
- 구조화 조회(수치형) + 답변 생성(근거 인용, 모르면 모른다).
- REST/SSE API.

### Phase 3 — 프론트엔드 (1.5주)
- 채팅 UI + 예시 질문 칩 + 출처 표시 + 근거 그래프 패널.

### Phase 4 — 평가·튜닝 (1주)
- 제공된 예시 질의 + 변형 질의로 **골든 테스트셋** 구성.
- 정확도(특히 수치)·근거 적합성·환각률 측정, 라우팅/프롬프트 튜닝.

### Phase 5 — 운영 전환 준비 (후속, 범위 외 MVP)
- 사내 SSO/접근제어, 질의 로깅 대시보드, 월별 운영기준 자동 재인덱싱(스케줄러).

> 예상 총 기간: **약 6.5~7주** (1인 기준, 데이터 안정 가정)

## 8. 평가 계획 (검증 단계 필수)

- **골든셋**: 제공된 5개 예시 + 각 5~10개 변형(동의어·축약·오타 포함) = 50~100문항.
- **지표**: 수치 정확도(Exact match), 근거 출처 정확도, 환각률(근거 없는 주장 비율), "모름" 적정성.
- **회귀 테스트**: 인덱스/프롬프트 변경 시 골든셋 자동 재실행.
- 수치형은 100% 정확 또는 "확인 필요" 표기를 목표(상담 사고 방지).
